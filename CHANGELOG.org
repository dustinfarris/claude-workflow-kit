#+title: Changelog — workflow-kit

* 0.5.0 — [2026-07-12]

Six-item patch from real-project friction, dependency-ordered. PRD Success Criteria are now a numbered list (~templates/PRD.org~, with a comment that the numbers are stable handles — never renumbered, an amendment retires rather than reassigns); create-prd states the numbering; user-stories' AC drift guard traces an AC's intent to a specific PRD Success Criterion cited as SC-n rather than the section in general. This was the prerequisite for the next fix: an extras promote-design run had cleared brainstorm/PRD alignment via provenance phrasing ("the PRD was created from it, so no divergence") that a human had to re-derive by hand to trust — a compliance claim indistinguishable from its own absence isn't a check. promote-design's closing step now requires the demonstration itself before DESIGN.org goes up for approval: a criterion-by-criterion SC-n → DESIGN section trace, plus a disposition line (faithful copy / rejected-alternative D-entry / dropped) for every brainstorm position the PRD overrode, with an example block; absence of the trace counts as the check not having happened, at every weight class. Input resolution stopped guessing too: promote-design's "newest under ~docs/superpowers/specs/~" mtime fallback — silently wrong under concurrent initiatives, and perturbable by an unrelated ~mv~ — is replaced with "list candidates and ask," mirrored into create-prd so both skills share one posture; no "newest" language remains in either skill's input resolution.

Brainstorm sessions genuinely drift into design altitude before a PRD exists (one real run produced a 250-line brainstorm document with schema and module names already decided) — survivable downstream since promote-design and the PRD filter absorb it, but confident pre-contract material is still a smell worth fencing. Superpowers owns the brainstorming skill itself, so the kit ships the fence for a human to paste instead of editing that skill: README gains a "Brainstorm kickoff fence" section holding the main thread at outcome altitude and parking mechanism material in a "for the design doc" section; create-prd gains one sentence naming that section as ordinary feedstock for promote-design, not something it extracts.

user-stories' single-open-batch rule was protective-only with no escape hatch: real evidence surfacing mid-batch (a bear-cub story invalidating a prior deferral) had no sanctioned path but the STOP. The rule now has exactly one exception — explicit human authorization for the run, quoted back to the human, with a LOGBOOK note on the batch heading recording the insertion and its story link, the evidence motivating it, and the authorization — while the original STOP sentence for the unauthorized case is untouched, still the only path reached when that authorization is absent.

Last, the Definition of Done's verification item and phase-close step 1 assumed a fixed mix invocation (~mix check~ for work-grade; ~mix format --check-formatted~ and ~mix test~ for personal-mvp) — bear-cub's real gate is ~mix precommit~, papered over with an ad hoc CLAUDE.md sentence rather than a first-class convention. Both project-setup templates now carry an optional ~Verify command:~ line under Stack & commands, defaulted to whichever invocation each weight class already ran so existing repos without the line are unaffected; the DoD blocks in user-stories, phase-close step 1, and story-closeout's Definition-of-Done evidence step all read that declared line and name its default instead of hard-coding a command.

* 0.4.2 — [2026-07-12]

Org inline-markup convention. Model-authored chain documents defaulted to =verbatim= for all inline monospace, ignoring org's semantic distinction between code and literal strings. org-conventions gains an Inline markup rule alongside the timestamp rule: =~code~= for anything a language would parse (identifiers, module/function names, expressions, shell commands, file paths), =verbatim= only for literal non-code strings quoted exactly (stored values, config keys, log output); prefer =~code~= when unsure. Audited the kit's own shipped skills, templates, and agents against the new rule; found one violation, in the Advisories example block inside org-conventions/SKILL.md itself — =async: true=, =ecto_sqlite3=, and =busy_timeout= were marked =verbatim= where they're code. Corrected the markup delimiters only, wording untouched, honoring the block's crown-jewel protection in CLAUDE.md. No sweep of existing project chain documents; story excerpts stay verbatim-faithful to the DESIGN they quote, markup warts included.

* 0.4.1 — [2026-07-11]

Brainstorm-document lineage rules. The brainstorm document is consumed by both promotion skills but predates the PRD, and real brainstorms produce design-altitude material — two gaps followed from that. First, promote-design's "copy the spec's design content faithfully" instruction made the pre-contract document de facto authoritative over human pivots made at PRD review; step 3 now states that the approved PRD.org outranks the brainstorm document, conflicting material is surfaced rather than copied, and a real-judgment resolution mints a design-time Decision Log entry whose rejected alternative is the brainstorm document's superseded position — faithful copying remains the rule for everything the PRD doesn't touch. Second, neither skill told the session that spec/PRD divergence is expected rather than an error to repair; both skills' Input sections now carry a lineage note: the brainstorm document predates the PRD, a human pivot at PRD review is expected and healthy, and PRD.org/DESIGN.org supersede it on contact. Terminology cleanup alongside: both skills now say "the brainstorm document" in prose instead of "spec" (the docs/superpowers/specs/ lookup path is superpowers' own convention and is unaffected).

CANCELLED (2026-07-11): teaching create-prd to reuse a pre-existing initiative directory or prefer in-directory brainstorm documents. Declined — the ruling went the other way: brainstorm outputs stay out of initiative directories entirely; a repo-CLAUDE.md convention line owns this, not skill logic.

* 0.4.0 — [2026-07-11]

promote-prd renamed to create-prd. The prd-lock hook is approval-blind: it unconditionally blocks Write/Edit/MultiEdit on any =*PRD.org= path and write-shaped Bash (redirects, =sed -i=, =tee=, =mv=/=cp= onto, =rm=, =truncate=) targeting =PRD.org=, so the skill's own "write PRD.org" step would have been blocked by the kit's own hook on first live use, and the revision loop was impossible once the file existed. Fixed: the skill now writes only to =PRD-draft.org= — a filename that matches neither the hook's filename glob nor its Bash regex, confirmed against hooks/prd-lock.sh directly — and iterates on the draft with the human without restriction. Approval is now the human's own rename, =mv .../PRD-draft.org .../PRD.org=, run in their own shell; from that moment the file sits under prd-lock and is the locked contract (a draft-then-rename flow). promote-design gains a guard: if the initiative directory holds PRD-draft.org but no PRD.org, STOP rather than read the draft as if it were the approved contract. hooks/prd-lock.sh itself is unchanged; hooks_test.sh gains cases for drafts passing the hook and mv/cp onto PRD.org still blocking. Renamed repo-wide: skill directory, frontmatter, README pipeline table, both project-setup CLAUDE.md variants, plugin.json description.

* 0.3.4 — [2026-07-11]

The conversation-born-terms invariant in CLAUDE.md now explicitly covers provided wording — an item scoped for the 0.3.2 patch but missed when it shipped. A verbatim block handed in via a prompt can smuggle a coined handle into spec text just as easily as session-drafted prose, so provided wording gets the same term audit before it ships. Doc-only change; no hook or skill behavior touched.


* 0.3.3 — [2026-07-11]

friction.org retired from the iteration loop. Its gate-and-cite format is replaced by a local, gitignored todo.org for personal backlog tracking — not a repo convention, so it isn't named in checked-in docs. CLAUDE.md's Iteration loop and Testing sections and README.org's Iteration loop bullet now read "kit edit (from real project use) → CHANGELOG.org entry → version bump," dropping the friction-entry gate; edits are justified by an explicit CHANGELOG rationale instead. friction.org's five DONE entries were already reflected in past CHANGELOG entries; its two open TODOs (initiative-close skill, CLAUDE.md-declarable verify command) carried forward into the local todo.org, not into the repo. (Superseded [2026-07-13]: todo.org promoted to the checked-in ~docs/backlog.org~ registrar file, reformatted to the dated-heading registrar shape; it is now a repo convention.)


* 0.3.2 — [2026-07-11]

Advisories gain an Issuance rule, closing the gap left by the payload litmus: the litmus said what an advisory must contain, not when one should be written at all. An advisory is now issued only where a superseded position could trap a future reader — never as a mutation record; additive or neutral body edits get no entry, since git and the gate's reconciliation pass already own mutation completeness. A "Sections affected" line may name a section only if that section's superseded position is stated in the payload — naming an affected section without stating what it used to say is the litmus failure in disguise. Ratified by human ruling, added verbatim to org-conventions' Advisories section.


* 0.3.1 — [2026-07-11]

Three fixes from the bear-cub shakedown. Timestamps unified: org-conventions' two competing formats (LOGBOOK inactive timestamps vs. Advisory/gate-record =%Z= timestamps) collapse into one — every bracketed timestamp in chain documents (LOGBOOK notes, gate records, Advisory headings) is now an org-native inactive timestamp via =date '+%Y-%m-%d %a %H:%M'=; the =%Z= bullet is gone, and both canonical Advisory examples updated to match. D-entry dates like =(2026-07-10)= remain inline notation, unaffected. user-stories' AC drift guard now traces inward as well as outward: a requirement the embedded excerpt enumerates for the story's scope must be covered by an AC or surfaced as uncovered, not silently dropped. It also gains a design-silence clause — where the design doesn't state a detail an AC would need, write the AC at the design's specificity or surface the gap with a =# GAP:= comment, rather than inventing the missing detail. hooks_test.sh gains a =%Z=-string tripwire over skills/templates, mirroring the existing Changelog tripwire.


* 0.3.0 — [2026-07-11]

Two-log structure, batch/gate ontology, phase-close slimmed to gate mechanics. The single =* Changelog= is replaced by two headings with distinct referents: =* Decision Log= (append-only, Dn-numbered judgments about the system being built) and =* Advisories= (superseded positions preserved for cold readers of the document), cross-cited by D-number. org-conventions defines both, with the payload litmus (FROM-first) and the paired invariants — an Advisory without a body edit, and a post-canon D-entry without a paired Advisory, are both suspect. Phases are demoted to batches: a batch is the unit that ends at a gate, "Phase N" is an optional label, and the gate record (a LOGBOOK entry on the batch heading) doubles as the closed-marker — the open batch is the one without a gate record. PLAN.org carries batch headings; user-stories appends a new batch and refuses to open one over an unclosed batch. phase-close is rewritten to gate mechanics only — verification (plus PRD-declared gate criteria), behavior/PLAN audit that routes repairs back through update-design, advisory withdrawal/promotion, and the gate record; its old v2-candidate promotion and stages-earned-their-keep retrospective move to the (not-yet-built) initiative-close skill. update-design's pivot handling becomes a three-step (mint D-entry → body edit → Advisory). New CLAUDE.md invariant: conversation-born terms don't ship except as parenthetical mnemonics after the rule they compress. Stale =references/org-conventions.md= paths corrected to the skill. hooks_test.sh gains a Changelog-string tripwire over skills/templates/project-setup.
