#+title: Friction Log — workflow-kit

Every kit edit starts here: friction observed in real project use → entry below → kit edit → CHANGELOG entry → version bump. Edits without a friction entry are suspect (see CLAUDE.md). Entries also record friction that was considered and deliberately NOT acted on, so the same idea isn't relitigated.

Entry format:

#+begin_example
* TODO [project] one-line description of the friction
:LOGBOOK:
- Note taken on [YYYY-MM-DD Day HH:MM] \\
  What happened, which stage/skill/hook, and what the fix candidate is.
:END:
#+end_example

Mark =DONE= when the fix ships (cite the CHANGELOG version) or =CANCELLED= with the reason it was declined.

* DONE [doctrine] "Changelog" conflated document-mutation record with system-judgment registry
:LOGBOOK:
- Note taken on [2026-07-11 Sat 09:58] \\
  The single `* Changelog` had two incompatible referents: a changelog's referent is the document (a mutation record for stateless readers — what the doc used to say), a decision log's is the system being built (a judgment registry — which fork was taken and why). One heading could not serve both without each corrupting the other. Resolved in 0.3.0 as a two-log structure: `* Decision Log` (append-only, Dn-numbered judgments) + `* Advisories` (superseded positions for cold readers), cross-cited by D-number, with a payload litmus (FROM-first) and paired invariants so drift is detectable from both directions. Shipped 0.3.0.
:END:
